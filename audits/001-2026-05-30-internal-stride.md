## [Informational] `take_intent_partial` multi-take — investigated, not exploitable

**Severity:** Informational &nbsp;|&nbsp; **Status:** Resolved (false positive) &nbsp;|&nbsp; **Category (STRIDE):** Tampering

**Location:** `sources/settlement.move:103` (`take_intent_partial`),
`mark_intent_settled` at `sources/settlement.move:276`.

**Hypothesis.** `take_intent_partial` does not mark the intent settled (only `settle_*` calls
`mark_intent_settled`). The concern was that a solver could `take_intent_partial` the same intent
twice before settling — the `!is_intent_settled` guard still passes on the second take — then settle
both receipts and double-count `settled_intent_count` / score.

**Why it is not exploitable.** Both receipts are hot potatoes (no `drop`), so both MUST be consumed
in the same PTB. Each `settle_*` ends in `mark_intent_settled`, which does
`intent_settled.insert(id)`. `sui::vec_set::insert` aborts with `EKeyAlreadyExists` on a duplicate
key, so the second settle aborts and the entire PTB reverts atomically — no double-count is ever
committed.

```text
take_partial(id, 400)  -> (c1, r1)   // ok, not yet settled
take_partial(id, 400)  -> (c2, r2)   // ok, still not settled
settle(r1)             // mark_intent_settled inserts id      -> ok
settle(r2)             // mark_intent_settled inserts id again -> abort, PTB reverts
```

Across separate transactions it is also blocked: a receipt cannot outlive its PTB, so the first
take must settle in the same tx (marking the intent), and any later take aborts with `EAlreadySettled`.

**Evidence.** Test `flow_tests::test_double_partial_settle_same_intent_reverts` exercises the
double-take + double-settle path and confirms the revert.

**Residual note.** The duplicate-insert + hot-potato combination is load-bearing for this safety
property. Keep `mark_intent_settled` using a set that aborts on duplicate; do not switch to an
idempotent insert without re-introducing an explicit single-settle guard.

## [Medium] Expired winning intent re-queued indefinitely

**Severity:** Medium &nbsp;|&nbsp; **Status:** Open &nbsp;|&nbsp; **Category (STRIDE):** Denial of Service

**Location:** `sources/settlement.move:83` / `:115` (`take_*` expiry check),
`sources/settlement.move:424` (`trigger_fallback` re-queue).

**Description.** If a winning intent expires between selection and settlement, `take_*` aborts with
`EIntentExpired`, the solver misses the deadline, and `trigger_fallback` re-queues the intent into
the next epoch. The intent is still expired next epoch, so `take_*` fails again and it is re-queued
again — an indefinite loop. The assigned solver may also be slashed for a timeout outside its control.

**Impact.** User funds locked in a perpetually-requeued expired intent; unfair slashing.

**Recommendation.** In `trigger_fallback`, refund expired intents to their owner (cancel) instead of
re-queueing. Optionally exclude provably-expired intents from slashing attribution.

## [Medium] Unbounded loops enable gas DoS

**Severity:** Medium &nbsp;|&nbsp; **Status:** Open &nbsp;|&nbsp; **Category (STRIDE):** Denial of Service

**Location:** `sources/settlement.move:319` (`close_batch` pair loop), `:368`
(`distribute_rewards` solver loop), `:402` (`trigger_fallback` intent loop).

**Description.** These loops iterate over committed pairs, winning solvers, and winning intents.
Their sizes derive from the winning allocation, which an auctioneer composes. A maliciously large
allocation could push `close_batch` past the transaction gas limit so the batch can never close.

**Impact.** Epoch cannot be closed → settlement liveness failure for that epoch.

**Recommendation.** Cap the number of bids/pairs per allocation in `auction::submit_allocation`,
and document the per-epoch maximum batch size.

## [Low] Overpaid protocol fee not refunded

**Severity:** Low &nbsp;|&nbsp; **Status:** Open &nbsp;|&nbsp; **Category (STRIDE):** Tampering

**Location:** `sources/settlement.move:333-335`.

**Description.** `close_batch` asserts `fee.value() >= required_fee`, then deposits the entire coin
via `treasury::deposit_fee`. Any excess above `required_fee` is absorbed into the treasury with no
refund to the caller.

**Impact.** Caller may over-donate; no fund loss to users, but unexpected for integrators.

**Recommendation.** Split `required_fee` off the coin and return the remainder to the caller, or
document that the full coin is consumed.

## [Informational] Score normalization trusts live DeepBook mid

**Severity:** Informational &nbsp;|&nbsp; **Status:** Acknowledged &nbsp;|&nbsp; **Category (STRIDE):** Tampering

**Location:** `sources/settlement.move:177` (`read_mid_price` in `settle_intent`).

**Description.** Surplus normalization uses the DeepBook mid price at settlement time. A thin-liquidity
numeraire pool could be manipulated to inflate normalized score. User payout is unaffected (EPSR and
benchmark floor are verified on raw token units), so this affects auction objective and rewards only.

**Recommendation.** Restrict numeraire pools to deep, allowlisted markets; add depth/age checks
(`price_oracle_max_age_ms` already exists in config). Tracked in the mechanism spec risk register.

## [Informational] `settle_*` does not re-check `sender == receipt.solver`

**Severity:** Informational &nbsp;|&nbsp; **Status:** Acknowledged &nbsp;|&nbsp; **Category (STRIDE):** Spoofing

**Location:** `sources/settlement.move:146` / `:161`.

**Description.** `receipt.solver` is set at `take` to the taker. `settle_*` does not require the
settler to equal `receipt.solver`. Since the receipt is a hot potato confined to one PTB, a third
party cannot obtain it independently; at most the original taker can hand it to another call in the
same PTB, which only shifts reward attribution, not funds.

**Recommendation.** Optional: assert `ctx.sender() == receipt.solver` in `settle_*` for clarity.

## [Medium] Zombie shared object after full drain via partial fills

**Severity:** Medium &nbsp;|&nbsp; **Status:** Open &nbsp;|&nbsp; **Category (STRIDE):** Denial of Service

**Location:** `sources/intent_book.move:186` (`consume_intent_partial`).

**Description.** `consume_intent_partial` takes `&mut Intent` and never deletes the object. When a
sequence of partial fills drains the balance to zero, `fill_status` is set to `Filled` but the
shared object persists with a zero balance, and `take_intent_partial` re-queues it for the next
epoch. The drained intent lingers as state and its storage rebate is never reclaimed.

**Impact.** State bloat; a zero-balance intent re-enters batches and may be taken/settled as a
no-op next epoch. Storage rebate for the drained object is never reclaimed.

**Recommendation.** When the last partial fill drains the balance, delete the object (full consume
path) and do not re-queue.

## [Low] `update_intent_params` must reuse `can_modify_intent` gating

**Severity:** Low &nbsp;|&nbsp; **Status:** Open &nbsp;|&nbsp; **Category (STRIDE):** Tampering

**Location:** `sources/intent_book.move:144` (`update_intent_params`), expected wrapper in
`sources/auction.move`.

**Description.** `update_intent_params` lets the owner change `min_amount_out` and `target_epoch`
arbitrarily. `cancel_intent`'s wrapper already gates with `can_modify_intent` (blocking changes to
an intent already in the winning set). The update path must apply the same gate; otherwise an owner
could mutate a winning intent's minimum after selection and grief the assigned solver.

**Recommendation.** Ensure the `auction` update wrapper calls `can_modify_intent` before delegating,
identically to `cancel_intent`.

## Methodology

STRIDE applied per public/package function: Spoofing, Tampering, Repudiation, Information
Disclosure, Denial of Service, Elevation of Privilege. Each entry point was checked for caller
authorization, phase gating, balance conservation, integer rounding direction, event emission, and
loop bounds. Findings above are the deltas from "defended"; categories not listed for a function
were judged adequately mitigated at this commit.
