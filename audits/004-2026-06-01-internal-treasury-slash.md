## Background

Since review 003 the protocol treasury was extended to also custody **slashed stake**:
`ProtocolTreasury<N, Stake>` gained `stake_balance: Balance<Stake>`, `total_stake_slashed`, and
`total_fallback_bounty_paid`, and `settlement::trigger_fallback` now routes confiscated bonds through
it (bounty split to the caller + `deposit_slashed_stake` for the remainder). This is the first review
covering that change, the `solver_registry` bond/slash accounting, and a re-audit of `settlement`
with the treasury + registry threaded through `close_batch` / `trigger_fallback`.

**Trust-boundary note.** `reserve_stake` / `release_stake` / `slash_reserved_stake` /
`record_settled` in `solver_registry` and `deposit_*` / `withdraw_reward` / `record_fallback_bounty`
in `treasury` are all `public(package)` — reachable only through `auction` / `settlement`. The user-
facing registry entries (`register_solver`, `top_up_stake`, `withdraw_available_stake`,
`deregister_solver`) enforce sender ownership and the `reserved_stake ≤ stake` invariant; `init_*` and
`reactivate_solver` require `AdminCap`. The findings below are the deltas from "defended".

## [Medium] Suspend-threshold evasion via deregister + re-register

**Severity:** Medium &nbsp;|&nbsp; **Status:** Open &nbsp;|&nbsp; **Category (STRIDE):** Elevation of Privilege / Repudiation

**Location:** `sources/solver_registry.move:83` (`register_solver`), `:131` (`deregister_solver`),
`:194-195` (`slash_reserved_stake` suspend logic).

**Description.** `SUSPEND_THRESHOLD = 3` auto-suspends a solver once `slash_count` reaches 3 — the
core anti-griefing penalty. But `slash_count` and `status` live entirely inside the per-solver
`SolverInfo`, which `deregister_solver` **removes** from the table:

```move
let SolverInfo { .. } = registry.solvers.remove(solver);   // slash history destroyed
```

`register_solver` then re-creates a fresh record for the same `ctx.sender()`:

```move
SolverInfo { url, total_settled: 0, slash_count: 0, status: SolverStatus::Active, reserved_stake: 0 }
```

Once a suspended solver's `reserved_stake` reaches 0 (all obligations resolved), it can
`deregister_solver` (no `AdminCap`, only `EReservedStakeOutstanding` gates it), reclaim its full
remaining stake, and immediately `register_solver` again from the **same address** — `slash_count`
back to 0, `status` back to `Active`. The suspension is lifted permanlessly and the only friction is
re-staking the minimum (which it gets back). `reactivate_solver` (the intended, `AdminCap`-gated
recovery path) becomes redundant.

**Impact.** The suspend control is fully bypassable. A repeatedly-faulting / griefing solver can stay
active indefinitely, defeating the reputation mechanism that protects settlement liveness.

**Recommendation (hướng sửa).** Persist slash reputation across deregistration. Add a separate
`Table<address, u64>` (or `Table<address, SlashRecord>`) that `deregister_solver` does **not** clear;
on `register_solver`, re-seed `slash_count`/`status` from it (a previously-suspended address comes
back Suspended, or is refused re-registration without `AdminCap`). Regression: register → 3 slashes →
deregister → register → assert solver is still `Suspended` and `slash_count` is carried.

## [Medium] Protocol treasury funds are permanently locked (no withdrawal path)

**Severity:** Medium &nbsp;|&nbsp; **Status:** Open &nbsp;|&nbsp; **Category (STRIDE):** Denial of Service

**Location:** `sources/treasury.move` (whole module — only exit is `withdraw_reward` at `:72`).

**Description.** The treasury holds two assets: numeraire fees (`balance: Balance<N>`) and confiscated
stake (`stake_balance: Balance<Stake>`). The **only** way value leaves the object is
`withdraw_reward` (`public(package)`), called by `settlement::distribute_rewards` to pay solver
rewards out of the **numeraire** balance. There is:

- **no** withdrawal for `stake_balance` — every bond slashed in `trigger_fallback` via
  `deposit_slashed_stake` accumulates forever; and
- **no** admin/governance withdrawal for residual numeraire fees (fees collected in excess of the
  rewards ever paid out).

`init_treasury` is `AdminCap`-gated but no `AdminCap` function can move funds out afterward.

**Impact.** All confiscated stake and net protocol revenue are unrecoverable — locked in the shared
object for the lifetime of the package. For mainnet this is a permanent loss of protocol income and
of every slashed bond.

**Recommendation (hướng sửa).** Add `AdminCap`-gated `withdraw_protocol_fees<N, Stake>(...)
: Coin<N>` and `withdraw_slashed_stake<N, Stake>(...): Coin<Stake>`, each emitting a withdrawal
event, and (per the open "AdminCap → multisig/timelock" item) routed through that capability.
Coordinate with finding F-004-8: do **not** decrement `total_stake_slashed` on withdrawal — it is a
cumulative gross metric, so introduce a separate `stake_balance`-tracked accounting if a net figure
is needed. Tests: deposit fee + slash → admin withdraws → balances reconcile.

## [Medium] Unbounded loops enable gas DoS (carried from 001)

**Severity:** Medium &nbsp;|&nbsp; **Status:** Open &nbsp;|&nbsp; **Category (STRIDE):** Denial of Service

**Location:** `sources/settlement.move:335` (`close_batch` pair loop), `:390`
(`distribute_rewards` solver loop), `:459` (`trigger_fallback` winner-intent loop).

**Description.** First raised in [001](001-2026-05-30-internal-stride.md) and still unaddressed. The
loops iterate over committed pairs, winning solvers, and winning intents — sizes derived from the
auctioneer-composed winning allocation. A maliciously large allocation can push `close_batch` (or the
permissionless `trigger_fallback`) past the transaction gas limit, so the epoch can never leave
Settlement. The treasury rework adds per-intent slash + deposit work inside the `trigger_fallback`
loop, so the surface is unchanged-to-slightly-worse.

**Impact.** Settlement liveness failure for an epoch that cannot be closed *or* failed.

**Recommendation (hướng sửa).** Cap bids/pairs/intents per allocation in
`auction::submit_allocation`, and document the per-epoch maximum batch size. Same as 001/003.

## [Low] Object-binding: treasury/registry not pinned to canonical IDs

**Severity:** Low &nbsp;|&nbsp; **Status:** Open &nbsp;|&nbsp; **Category (STRIDE):** Spoofing / Tampering

**Location:** `sources/settlement.move` — `close_batch` (`:312`), `settle_intent*` (`:156` / `:171`),
`trigger_fallback` (`:439`).

**Description.** Settlement entries trust whatever `SolverRegistry<Stake>` and
`ProtocolTreasury<N, Stake>` shared objects the caller passes. The only instance check is
`assert_treasury_numeraire<N>` — a **type** check, not an identity check. `AuctionState` and
`GlobalConfig` are publish-time singletons, but the registry and treasury are created by admin via
`init_registry` / `init_treasury` and nothing enforces a single instance or binds their IDs in
config. If more than one matching instance ever exists, a permissionless caller of `close_batch` /
`trigger_fallback` could pass:

- a **fatter** treasury (same numeraire) → `distribute_rewards` pays larger rewards to the winning
  solvers (potentially the caller); or
- an **empty / foreign** registry → `release_stake` and `record_settled` silently no-op (reservations
  not found is a no-op in `release_stake`), so the real bonds are never released and metrics never
  update, while settlement still marks intents settled in the singleton `AuctionState`.

**Impact.** Conditional on admin creating multiple instances (operational footgun rather than an
unconditional exploit), but the trust assumption is implicit and undocumented.

**Recommendation (hướng sửa).** Store canonical `registry_id` / `treasury_id` in `GlobalConfig` (set
once by admin) and assert `object::id(registry) == config.registry_id()` etc. in the settlement
entries; or explicitly document and operationally enforce the single-instance invariant.

## [Informational] Fallback bounty payable to the at-fault solver

**Severity:** Informational &nbsp;|&nbsp; **Status:** Open (latent) &nbsp;|&nbsp; **Category (STRIDE):** Repudiation

**Location:** `sources/settlement.move:489-497` (`trigger_fallback` bounty path).

**Description.** `trigger_fallback` is permissionless and pays `fallback_bounty_bps` of the total
slashed stake to `ctx.sender()`, with no exclusion of the slashed solvers. A solver that is being
slashed (or a colluder) can call `trigger_fallback` itself and claw back a fraction of its own slash.
Inert today because `fallback_bounty_bps` defaults to 0 (`MAX_FALLBACK_BOUNTY_BPS = 1000`), but
becomes a real partial-refund-to-the-guilty if the bounty is ever raised.

**Recommendation (hướng sửa).** If/when `fallback_bounty_bps > 0`, exclude the slashed owners from
the bounty (or pay the bounty only to a caller not in the slashed set), or document that the bounty
is paid to whoever triggers fallback regardless of fault.

## [Informational] `auctioneer_share_bps` / `auctioneer_reward_cap` are dead config

**Severity:** Informational &nbsp;|&nbsp; **Status:** Open &nbsp;|&nbsp; **Category (STRIDE):** —

**Location:** `sources/config.move:96-97`, `:242-249`, `:343-345`; expected consumer
`sources/settlement.move:371` (`distribute_rewards`).

**Description.** `auctioneer_share_bps` and `auctioneer_reward_cap` are declared, defaulted, settable
(`AdminCap`), and readable, but **never consumed** anywhere. `distribute_rewards` pays only
`winning_solver_list` (the solvers credited in `record_settlement`); in the benchmark-fallback path
the auctioneer is recorded *as* the solver and is therefore rewarded through the solver path, not via
these params. An operator could set `auctioneer_share_bps` expecting it to take effect — it does not.

**Recommendation (hướng sửa).** Either implement a distinct auctioneer reward consuming these params,
or remove them to avoid an inactive-control footgun.

## [Informational] `total_stake_slashed` accounting split across two functions (fragile)

**Severity:** Informational &nbsp;|&nbsp; **Status:** Open &nbsp;|&nbsp; **Category (STRIDE):** —

**Location:** `sources/treasury.move:64-70` (`record_fallback_bounty`), `:53-62`
(`deposit_slashed_stake`); caller `sources/settlement.move:489-505`.

**Description.** In `trigger_fallback`, `record_fallback_bounty` adds the `bounty` portion to
`total_stake_slashed` (and to `total_fallback_bounty_paid`), while `deposit_slashed_stake` adds the
remaining `(total − bounty)` to `total_stake_slashed`. The net is correct today —
`total_stake_slashed` ends at the gross slashed amount, and the invariant
`stake_balance == total_stake_slashed − total_fallback_bounty_paid` holds **as long as no stake is
ever withdrawn**. The coupling is non-obvious: `record_fallback_bounty` mutating `total_stake_slashed`
for value that leaves the treasury is surprising, and the invariant silently breaks the moment a stake
withdrawal (finding F-004-2) is added without accounting care.

**Recommendation (hướng sửa).** Document the invariant explicitly, and prefer accounting the gross
slashed amount **once** (e.g. at a single deposit call) so the metric does not depend on two callers
summing correctly.

## [Informational] `reactivate_solver` does not reset `slash_count` — by design

**Severity:** Informational &nbsp;|&nbsp; **Status:** Acknowledged (by design) &nbsp;|&nbsp; **Category (STRIDE):** —

**Location:** `sources/solver_registry.move:145` (`reactivate_solver`), `:194-195` (suspend check).

**Description.** After an `AdminCap`-gated `reactivate_solver`, `slash_count` is left at ≥
`SUSPEND_THRESHOLD`. Because the suspend check is `info.slash_count >= SUSPEND_THRESHOLD`, the *next
single slash* re-suspends the solver immediately.

**Decision (this review).** Intended — **strict probation**: a reactivated solver gets exactly one
more chance and is re-suspended on any further fault. No code change. Recorded here so it is not
mistaken for a bug in future reviews. (Note: this control is only meaningful once finding F-004-1 is
fixed — otherwise a solver bypasses it entirely by deregister + re-register.)

## Re-verified OK (no change required this round)

- **003 expired-intent fix holds.** `trigger_fallback` slashes and re-queues only intents with
  `now <= deadline`; intents that expired before settlement are neither slashed nor re-queued (owner
  reclaims via cancel). Confirmed against `settlement.move:464-480`.
- **Per-reservation slash dedupe is correct.** `slashed_reservations: VecSet<StakeReservationKey>`
  ensures a reservation shared by multiple intents in one bid is slashed at most once; remaining
  unsettled-but-valid intents on that reservation are still re-queued.
- **`close_batch` and `trigger_fallback` are mutually exclusive.** Both assert Settlement phase; the
  first to run transitions the epoch to Close / Failed, so the other aborts. No double-slash or
  double-reward across the two paths.
- **`close_batch` exact-fee split + remainder refund (003) is present and correct**
  (`settlement.move:349-355`).
- **Reservation invariant `reserved_stake ≤ stake.value()` is preserved** across `reserve_stake`
  (checks `available_stake_of ≥ amount`), `withdraw_available_stake` (caps at `available_stake_of`),
  and `slash_reserved_stake` (decrements both sides by `amount`). The defensive
  `amount <= stake.value()` assert in `slash_reserved_stake` cannot fail under this invariant.

## Methodology

STRIDE applied per public/package function across the in-scope modules: Spoofing, Tampering,
Repudiation, Information Disclosure, Denial of Service, Elevation of Privilege. Each entry point was
checked for caller authorization, capability gating, balance conservation (numeraire **and** stake),
the reserved-vs-total stake invariant, fund-exit reachability, slash attribution / dedupe, event
emission, and loop bounds. Findings above are the deltas from "defended"; categories not listed for a
function were judged adequately mitigated at this commit.
